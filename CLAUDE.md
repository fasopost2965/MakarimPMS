# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Projet

PMS Hôtel Makarim — projet interne (pas de logique SaaS multi-hôtels). Hôtel 3 étoiles, 24 chambres, Tétouan. Monorepo `backend/` (NestJS + Prisma + MySQL 8) + `frontend/` (React + Vite + TS + Tailwind + shadcn/ui), auth JWT (access + refresh), déploiement VPS Hostinger via Docker Compose + Nginx + Certbot.

## Commandes

### Backend (`cd backend`)
- Dev (watch) : `npm run start:dev`
- Build : `npm run build`
- Lint (fix) : `npm run lint`
- Tests unitaires : `npm run test`
- Tests e2e (vraie base MySQL, jamais de mock) : `npm run test:e2e`
- Un seul fichier e2e : `npx jest --config ./test/jest-e2e.json <fichier>.e2e-spec.ts`
- Un seul scénario e2e : `npx jest --config ./test/jest-e2e.json <fichier>.e2e-spec.ts -t "<nom du test>"`
- Migration Prisma : `npx prisma migrate dev --name <nom>`
- Reseed (rôles/permissions/données de démo) : `npx prisma db seed` — à relancer après toute migration ou après avoir touché `ALL_MODULES`/`rolesData` dans `prisma/seed.ts`
- Comptes de seed : mot de passe commun `Password123!`, emails du type `admin@makarim.test`, `reception@makarim.test`, `comptable@makarim.test`, `gouvernante@makarim.test`, `maintenance@makarim.test`, `rh@makarim.test` (voir `prisma/seed.ts`)

### Frontend (`cd frontend`)
- Dev : `npm run dev`
- Build : `npm run build` (= `tsc -b && vite build`)
- Lint (fix) : `npm run lint`

## Architecture backend

- Un module métier = un dossier sous `backend/src/modules/<domaine>/` avec `<domaine>.module.ts` / `.controller.ts` / `.service.ts` / `dto/`. Modules actuels (17, voir [docs/modules/MODULES_INDEX.md](docs/modules/MODULES_INDEX.md)) : `auth`, `rooms` (propriétaire exclusif de `Room`/`RoomType`/`RoomStatusLog`, module feuille sans dépendance, pas de controller propre — voir ci-dessous), `parameters` (propriétaire exclusif de `HotelConfig`/`TaxRateConfig`/`SeasonRate`/`CnssRateConfig` — module feuille, dépend uniquement de `audit` ; `billing`/`reservations`/le futur `hr` le consomment en façade, jamais Prisma direct sur ces tables), `reservations`, `stay` (renommé depuis `checkin` — `StayService`/`StayController`/`StayModule`, fichiers sous `backend/src/modules/stay/`), `housekeeping`, `maintenance`, `guests` (contient aussi `CompaniesController`/`CompaniesService` — Company reste une responsabilité du module guests, pas de module séparé), `billing` (Folio/FolioLine/Invoice), `payments` (`Payment`, dépend exclusivement de `billing`), `dashboard`, `audit`, `police` (`PoliceRecord`, registre légal des personnes hébergées — obligation DGSN ; dépend de `stay` en façade (`StayService.findOne`, jamais Prisma direct sur `Stay`) et de `audit` ; pas de clé de permission dédiée, réutilise `checkin:read`/`checkin:write` comme `companies` réutilise `guests:*`), `notifications` (F7 — CRM/marketing, canal email v1 ; `NotificationTemplate`/`NotificationLog` ; consommateur des évènements `reservation.confirmee`/`checkout.effectue` et d'un `@Cron` quotidien, dépend de `guests`/`reservations`/`stay` en façade et de `audit`), `self-checkin` (F6 — propriétaire exclusif de `SelfCheckinToken`, dépend de `guests`/`notifications` en façade), `booking-engine` (F4 — façade publique pure de `reservations`, aucune table propre), `document-ocr` (F5 — module feuille sans dépendance, aucune table Prisma propre, ne lit/n'écrit rien : extraction OCR consultative en mémoire uniquement). **Écart connu (déviation documentée)** : les routes HTTP (`/checkin/*`, `/checkout/:stayId`) et la clé de permission (`checkin:read`/`checkin:write`) du module `stay` sont volontairement restées nommées `checkin` — seuls les identifiants internes (dossier, classes, fichiers) ont été renommés, pour ne toucher ni le frontend (`frontend/src/features/checkin/` reste lui aussi nommé `checkin`, dette de nommage séparée) ni la suite e2e. Ne pas renommer `checkin:read`/`checkin:write` en `stay:*` sans une PR dédiée qui traite RBAC + routes + frontend ensemble.
- **RBAC** : `Permission` est une paire `(module, action)` en base (`module`/`action` sont des `String` libres, pas des enums — un nouveau module n'exige pas de migration de schéma). `@RequirePermission(module, action)` (decorator) + `PermissionsGuard` (global via `APP_GUARD`, exécuté après `JwtAuthGuard`) vérifient une requête Prisma fraîche à chaque appel — aucune permission n'est mise en cache dans le JWT, retirer un droit à un rôle prend effet immédiatement. `backend/prisma/seed.ts` (`ALL_MODULES`, `rolesData`) est la seule source de vérité des attributions rôle → permission. Exception : quand l'autorisation dépend du **contenu de la requête** (ex. `guests:blacklist` — seule une transition de catégorie vers/depuis `BLACKLIST` l'exige, pas les autres) le contrôle ne peut pas être exprimé par le décorateur statique et se fait par une requête `permission.findFirst` manuelle dans le service (voir `GuestsService.updateCategorie`). Écart connu : `docs/modules/rooms.md` §7 documente des clés dédiées `rooms:read`/`rooms:write`/`rooms:status` — non implémentées, `GET /rooms`/`PATCH /rooms/:id/statut` restent sur `HousekeepingController` sous `housekeeping:read`/`housekeeping:write` tant qu'aucune route de configuration de chambre n'existe.
- **Un seul chemin d'écriture par champ sensible** : chaque état métier a une méthode canonique unique qui l'écrit, journalise dans une table de log dédiée, et est réutilisée par tous les appelants (jamais dupliquée). Exemples : `RoomsService.transitionRoom` est le seul point d'écriture de `Room.statut` (écrit aussi `RoomStatusLog`) — `housekeeping`, `stay` et `maintenance` en dépendent tous, jamais l'inverse ; `GuestsService.updateCategorie` est le seul point d'écriture de `Guest.categorie` (écrit `GuestCategoryLog` + `AuditLog`). Tout nouveau champ métier sensible doit suivre ce pattern plutôt que d'être modifié directement via Prisma depuis plusieurs endroits.
- **Audit** (`backend/src/modules/audit/`) : `AuditService` est append-only (`writeLog()` + `findMany()` seulement, jamais d'update/delete). Toute opération sensible doit appeler `writeLog(tx, …)` **dans la même transaction Prisma** que la modification métier qu'elle audite — jamais après coup, jamais en asynchrone détaché (sinon un log peut silencieusement manquer). `tx` y est un paramètre obligatoire, pas optionnel.
- **Frontières de module** : un module ne doit jamais lire directement les tables d'un autre domaine via Prisma — il passe par le service exposé par le module propriétaire (`exports: […]` + import du module). Exemple : `GuestsService.factures()` appelle `BillingService.findInvoicesByGuestId()` plutôt que d'interroger `prisma.invoice` lui-même. De même, `PaymentsService.createPayment` n'écrit jamais dans `FolioLine` lui-même — il appelle `BillingService.creditFolioLine(folioId, montant, libelle, tx)` **dans la même transaction** que l'écriture du `Payment`, qui crée la ligne `PAIEMENT` créditrice (docs/modules/payments.md §2/§10 : `payments` ne dépend que de `billing`). De même encore, `BillingService.generateInvoice` (taux de TVA/taxe de séjour) et `ReservationsService.calculatePrixTotal` (grille tarifaire saisonnière) n'interrogent jamais `TaxRateConfig`/`SeasonRate` directement — ils appellent `ParametersService.getTaxRateMap()` / `ParametersService.getSeasonRatesForRoomType(roomTypeId, tx?)` (docs/modules/parameters.md §10 : `parameters` est un module feuille, tous les autres modules le consomment, jamais l'inverse). **Dérogation documentée** : `docs/modules/housekeeping.md` §11 interdit à `housekeeping` de dépendre de `reservations` (protection de la vie privée client), mais le rattrapage quotidien des statuts « relatifs à aujourd'hui » (voir plus bas) a besoin de savoir si une réservation arrive aujourd'hui — faute de cron, ce recalcul ne peut s'exécuter qu'à la lecture. `HousekeepingService` accède donc à `Reservation`/`Stay` exclusivement via les façades en lecture seule `ReservationsService.findConfirmedArrivingToday()` / `StayService.findActiveStayForRoom()` (jamais de Prisma direct) — c'est un écart assumé au référentiel gelé, pas un oubli.
- **Paramètres/configuration** (`backend/src/modules/parameters/`) : `HotelConfig` est un singleton dont l'`id` n'est **jamais** supposé égal à 1 (l'auto-incrément MySQL ne se réinitialise pas après les `deleteMany()` répétés du seed) — toujours le résoudre via `findFirst()`, jamais une constante codée en dur. Toute mutation (`updateHotelConfig`, `updateTaxRate`, `createSeasonRate`, `updateSeasonRate`, `removeSeasonRate`) exige un motif écrit (≥ 10 caractères) et écrit dans `AuditLog` dans la même transaction — réservé à `parameters:write` (Administrateur uniquement, RBAC plus restrictif que `billing:write`/`reservations:write` puisqu'il s'agit de configuration exceptionnelle, pas d'opérations métier quotidiennes). `CnssRateConfig` existe en base (readiness pour le futur module `hr`) mais n'a aucun endpoint tant que `hr` n'a pas de consommateur réel.
- **Paiements et solde de folio** : un règlement (`POST /payments`) crédite toujours un `folioId` (jamais seulement une facture — `invoiceId` reste optionnel, un acompte peut être encaissé avant qu'une facture n'existe). `computeSoldeDu` (`backend/src/modules/stay/utils/solde.ts`) est la seule fonction qui calcule un solde de folio, à partir des `FolioLine` (dont les lignes `PAIEMENT`, en négatif) — jamais à partir de la table `Payment` directement. Écart connu : `checkout()` (`StayService`) ne bloque pas encore si le solde cumulé est non nul (`BR-SEJ-004`/`INV-SEJ-002` non appliqués) — dette technique documentée, hors périmètre des modules `payments`/`stay` traités jusqu'ici.
- **Découplage événementiel** (`@nestjs/event-emitter`) : utilisé pour les transitions inter-modules qui doivent rester lâchement couplées (ex. `checkout.effectue`, écouté par `housekeeping` pour repasser la chambre à `A_NETTOYER`). Toujours `emitAsync` (jamais `emit`) quand l'appelant a besoin d'attendre l'effet du listener avant de répondre — **et aussi dès qu'un listener écrit en base une ligne qui référence par FK une entité dont l'appelant contrôle le cycle de vie** (constaté sur `reservation.confirmee` : un `emit` fire-and-forget avait laissé `NotificationsService.notify()` encore en vol après la suppression de la réservation par un test rapide, violation de contrainte FK `reservationId`). `emit` reste acceptable seulement pour un effet best-effort sans écriture dépendante (aucun cas de ce type dans le code actuel).
- **États relatifs à « aujourd'hui »** (ex. `RESERVEE`, `DEPART_PREVU`) : pas d'infrastructure de cron dans ce projet. Ces statuts sont recalculés de façon idempotente à chaque lecture pertinente (« rattrapage à la lecture », `HousekeepingService.reconcileDailyStatuses`), dans les deux sens (avancée et retour en arrière si la situation qui les justifiait disparaît).
- **Réutilisation/blacklist client** : `CreateReservationDto`/`WalkinCheckinDto` acceptent soit un `guestId` existant (déclenche `GuestsService.assertNotBlacklisted`), soit un objet `guest` pour en créer un nouveau (jamais `BLACKLIST` par défaut). `assertNotBlacklisted` est la seule règle bloquante associée à une catégorie client — les autres catégories (VIP/ENTREPRISE/AGENCE) sont de simples étiquettes.
- **Politique d'annulation & pénalités no-show** (`CancellationPolicy`, BR-RES-006) : propriété du module `reservations` (pas `parameters` — une politique d'annulation ne concerne que le cycle de vie de la réservation, contrairement à `TaxRateConfig`/`SeasonRate` consommés par plusieurs modules). `ReservationsService.remove()` (annulation) et `ReservationsService.markNoShow()` sont les deux seuls chemins d'écriture vers `ANNULEE`/`NO_SHOW` — `update()` (`PATCH /reservations/:id`) rejette explicitement ces deux statuts pour empêcher un contournement de `computeCancellationPenalty` (`reservations/utils/cancellation-penalty.ts`). **Écart documenté** : BR-RES-006 demande de « générer une ligne de folio de pénalité », mais une réservation annulée/no-show n'a jamais de `Stay`/`Folio` (ADR-002 : un `Folio` appartient toujours à un `Stay`) — le montant calculé est donc figé sur `Reservation.montantPenalite` (jamais recalculé après coup) plutôt que matérialisé en `FolioLine`. Son recouvrement (retenue sur un `ReservationDeposit` existant via `POST /reservations/:id/deposits/:depositId/rembourser`, ou facturation manuelle) reste une décision humaine de la réception/comptabilité, hors périmètre de ce calcul.
- **Restrictions tarifaires min stay / stop sale** (`RateRestriction`) : propriété du module `parameters` (comme `SeasonRate`, même consommateur `reservations`) — une ligne par type de chambre, chevauchements de périodes autorisés (contrairement à `SeasonRate` : plusieurs restrictions peuvent légitimement se superposer, la plus restrictive s'applique). `ReservationsService.assertRateRestrictionsSatisfied` (`reservations/utils/rate-restrictions.ts`) applique deux sémantiques distinctes, alignées sur la convention standard des moteurs de réservation hôteliers : `stopSale` bloque dès qu'**une seule nuit** du séjour tombe dans la période concernée (même en simple passage) ; `minStayNuits` ne s'applique qu'aux restrictions dont la période couvre **la date d'arrivée** (un séjour qui traverse une fenêtre de séjour minimum sans y arriver n'est pas concerné). Vérifié dans `create()` et dans la branche `datesOrRoomChanged` de `update()` (sinon un déplacement de réservation via `PATCH` contournerait la règle) — jamais dans le flux walk-in (`StayService.checkinWalkIn`), hors périmètre puisqu'un walk-in n'a pas de date d'arrivée future à restreindre.
- **Notifications CRM (F7, canal email v1)** : `NotificationsService.notify()` est le point d'entrée unique — jamais d'envoi direct ailleurs (consentement `Guest.consentementNotifications` et journalisation `NotificationLog` systématiques). Les listeners (`notifications/listeners/`) vivent dans le module **consommateur**, pas dans celui qui émet l'évènement — même convention que `housekeeping/listeners/checkout-effectue.listener.ts` : `ReservationsService.create()` émet `reservation.confirmee` (`emitAsync`, pas `emit` — voir la règle générale de découplage événementiel ci-dessus) sans jamais importer `NotificationsModule`, qui importe au contraire `ReservationsModule`/`StayModule`/`GuestsModule` en façade. Le rappel J-1 est un `@Cron('0 9 * * *')` (`notifications/cron/`, réutilise `ScheduleModule.forRoot()` déjà enregistré globalement par `HrModule` — pas besoin de le ré-importer). `MailerService` (SMTP via `nodemailer`) se dégrade silencieusement en simple `Logger.log` si `SMTP_HOST` n'est pas configuré — jamais d'exception pour une config manquante, seule une panne SMTP réelle fait échouer `NotificationsProcessor` (retry BullMQ, 3 tentatives).
- **Self check-in (F6, lien mobile pré-arrivée)** : `SelfCheckinToken` (un seul lien actif par réservation, `reservationId @unique`, régénéré = même ligne réécrite) expose deux routes `@Public()` (`GET`/`POST /self-checkin/:token`, throttlées 20/min — jamais de `@RequirePermission` dessus, `PermissionsGuard` laisse passer une route sans permission déclarée). **Écart documenté**, même famille que `Reservation.montantPenalite` (BR-RES-006) : `PoliceRecord` exige un `stayId` qui n'existe pas encore avant le check-in réel, donc les champs collectés (`numeroPiece`, `typePiece`, `dateNaissance`, provenance/destination) restent « en attente » sur `SelfCheckinToken` lui-même — jamais écrits dans `PoliceRecord` directement. Seuls les champs déjà présents sur `Guest` (`nom`/`prenom`/`telephone`/`email`/`nationalite`/`pieceIdentite`) sont mis à jour immédiatement via `GuestsService.update()`. La réception relit les champs en attente via `GET /reservations/:id/self-checkin-pending` (authentifié) pour pré-remplir `POST /police/:stayId` à l'arrivée — **jamais de dépendance `stay` → `self-checkin`** (StayService ne consomme rien de ce module), ce couplage reste un geste humain explicite, pas automatique. `generateLink()` réutilise `NotificationsService.notify()` (F7, `EvenementNotification.SELF_CHECKIN_LIEN`) pour l'envoi — aucun nouveau code d'envoi. Lien construit sur `FRONTEND_URL` (même variable que CORS) — **limite connue** : CORS reste restreint à cette unique origine (`main.ts`), donc une page self-checkin hébergée ailleurs échouerait aujourd'hui ; non résolu ici (même carve-out CORS que documenté pour le futur Booking Engine, F4, hors périmètre de cette itération).
- **Booking Engine public (F4, BR-RES-004)** : `BookingEngineModule` importe uniquement `ReservationsModule` — aucune logique de réservation dupliquée, `BookingEngineService` n'est qu'une façade (`GET /booking/availability` groupe `ReservationsService.checkAvailability()` par type de chambre et ajoute un prix indicatif via `ReservationsService.estimatePrixTotal()`, nouveau wrapper public de la méthode privée `calculatePrixTotal` ; `POST /booking/reservations` force `canal: DIRECT` et délègue à `ReservationsService.create()` telle quelle). **Anti-IDOR** : `CreatePublicReservationDto` n'a délibérément aucun champ `guestId` (seulement un objet `guest` pour un nouveau client) — combiné à `ValidationPipe({forbidNonWhitelisted: true})` global, toute tentative d'injecter un `guestId` est rejetée en 400, vérifié en live. La confirmation email (F7) part automatiquement, sans rien câbler dans ce module, puisque `ReservationsService.create()` émet déjà `reservation.confirmee` quel que soit l'appelant. **Course résiduelle acceptée** : entre la vérification de disponibilité et l'écriture, un conflit est rattrapé par la contrainte unique `RoomNight(roomId, date)` déjà éprouvée (`reservations-concurrency.e2e-spec.ts`) — pas de verrou applicatif dédié, le client public reçoit un 409 et retente, même logique que le flux interne. **Carve-out CORS** (`main.ts`) : un `CorsOptionsDelegate` (pas un objet statique) inspecte `req.url` pour distinguer les préfixes publics (`/api/booking`, `/api/self-checkin` — origine réfléchie, `credentials: false`, aucun cookie/Bearer requis) du reste de l'API (strictement `FRONTEND_URL`, `credentials: true`) — ferme rétroactivement la limite CORS documentée dans l'entrée F6 ci-dessus. Throttling dédié plus strict que le défaut global (`@Throttle`) : 30/min sur `availability`, 10/min sur `reservations` (scraping/spam de réservations).
- **Revenue Manager / Yield Management (F3, Phase 4 de `reporting`)** : `GET /reporting/yield-forecast` (`reporting:read`) — module `reporting` reste strictement read-only (INV-REP-001, `docs/modules/reporting.md`) : la recommandation est purement consultative, aucune écriture sur `SeasonRate`, la mise à jour effective de la grille tarifaire reste un acte humain via `parameters:write`. Pour chaque type de chambre et chaque jour de la plage demandée, calcule le taux d'occupation prévisionnel à partir de `RoomNight` (lecture Prisma directe, même convention que `FolioLine` dans `FinancialReportingService` — pas de service métier importé pour ce type de lecture cross-domaine) puis délègue le prix courant à `ParametersService.getSeasonRatesForRoomType()` (jamais de lecture directe de `SeasonRate`) via `calculateNightlyTotal()` réutilisé tel quel depuis `reservations/utils/pricing.ts` (précédent déjà établi par `billing`/`stay`, aucune duplication de la logique de tarification saisonnière). **Taux d'Occupation Net** (`docs/modules/reporting.md` §12) : le dénominateur de chambres exclut les chambres `EN_MAINTENANCE` du décompte, jamais comptées comme vendables — vérifié en live (mise en maintenance d'une chambre, `totalChambres` diminue immédiatement dans la prévision). Seuils de recommandation (`HAUSSE` ≥80% / `BAISSE` <40%, ajustement +15%/-10%) volontairement fixes dans `reporting/utils/yield-recommendation.util.ts`, pas administrables via `parameters` — dette technique documentée (`docs/modules/reporting.md` §16), hors périmètre de cette itération.
- **Scan CIN/Passeport OCR (F5)** : `POST /document-ocr/scan` (réutilise `guests:write` — même convention de réutilisation que `police`/`companies`, ce scan sert avant tout à préremplir la fiche client au check-in) accepte une image (JPEG/PNG/WebP, 8 Mo max, `multer` en `memoryStorage()`) et retourne les champs extraits de la zone MRZ (ICAO 9303, `document-ocr/utils/mrz-parser.ts`, fonction pure testée indépendamment de l'OCR) — **jamais de persistance de l'image** (aucune écriture disque, aucune table Prisma propre au module) et **jamais d'écriture directe sur `Guest`/`PoliceRecord`** : la réception relit et valide manuellement avant d'enregistrer via `GuestsService.update()`/`POST /police/:stayId` existants, même posture consultative que F3. `DocumentOcrService` réutilise un unique worker `tesseract.js` partagé entre requêtes (coût de chargement du modèle de langue). **Modèle de langue "eng" fourni en dépendance npm** (`@tesseract.js-data/eng`) plutôt que téléchargé depuis le CDN jsdelivr par défaut de tesseract.js au premier appel : évite toute dépendance réseau sortante au runtime (un VPS peut restreindre l'egress) et rend le comportement reproductible (verrouillé par `package-lock.json`) — `cacheMethod: 'none'` désactivé explicitement pour ne pas écrire une seconde copie décompressée sur disque (comportement par défaut de tesseract.js constaté en le testant : un fichier de 22 Mo apparaissait à la racine de `backend/`). Supporte les deux formats MRZ concernés par `TypePiece.CIN`/`PASSEPORT` : TD1 (CIN biométrique marocaine, 3 lignes de 30 caractères) et TD3 (passeport, 2 lignes de 44 caractères), avec validation des chiffres de contrôle ICAO 9303 (`checksumValide`) — un échec de checksum ne bloque jamais l'extraction (`avertissement` informatif seulement), une erreur OCR sur un champ ne doit pas priver la réception des autres champs correctement lus (vérifié en live : un nom mal lu par l'OCR n'empêche pas `checksumValide:true` ni l'extraction du reste des champs).
- **Pré-vérification de disponibilité pour le drag-and-drop planning (F8)** : `GET /reservations/availability` (`reservations:read`) — le déplacement lui-même n'a jamais eu besoin de nouveau code (`PATCH /reservations/:id` gérait déjà le changement de chambre/dates, re-verrouillage `RoomNight` et recalcul de prix, avant même cette itération) ; seul manquait un point de lecture pour griser une case cible invalide avant le drop. `ReservationsService.checkRoomAvailability` reste purement consultatif — `update()` reste le seul point d'écriture et l'arbitre final en cas de course résiduelle (même contrainte unique `RoomNight(roomId, date)` qu'ailleurs dans ce module). `excludeReservationId` (optionnel) exclut uniquement les propres nuits de la réservation déplacée — jamais les nuits liées à un `Stay` (client déjà en place, ne peut jamais être déplacé par un glisser-déposer) — vérifié en live : déplacer une réservation sur ses propres dates/chambre ne se signale plus en conflit avec elle-même, tandis qu'un chevauchement avec une **autre** réservation reste bien détecté. Réutilise `assertRateRestrictionsSatisfied` (B5, min stay/stop sale) en mode informatif seulement (`motifIndisponibilite`, jamais bloquant côté serveur) — vérifié en live avec une restriction stop-sale réelle.

## Architecture frontend

- `frontend/src/features/<domaine>/` : `api.ts` (appels `apiRequest<T>`), `types.ts`, `components/`, `pages/`. Pas de routeur (`react-router` absent) — navigation via un simple `useState<Tab>` + rendu conditionnel dans `App.tsx`.
- `lib/api-client.ts` (`apiRequest`) : intercepte les 401, tente un refresh de token, mutualise les refresh concurrents (une seule requête `/auth/refresh` même si plusieurs appels échouent en même temps), puis redirige vers l'écran de connexion si le refresh échoue aussi.

## Tests

- Toujours contre une vraie base MySQL, jamais de mock (`backend/test/*.e2e-spec.ts`).
- `backend/test/jest-e2e.json` a `maxWorkers: 1` et `testTimeout: 30000` — **ne pas retirer** (bcrypt sous parallélisme Jest casse le teardown : erreurs "unsupported charset" / "requiring a file after Jest environment torn down").
- Pour toute règle qualifiée de non-négociable, la couverture de test doit inclure une **preuve de rigueur sabotage/restore** documentée en commentaire : casser temporairement le mécanisme testé, confirmer que le test échoue bien de la manière attendue, restaurer, revérifier le vert. Un test qui reste vert après sabotage n'est pas discriminant (vérifier qu'aucune autre couche indépendante ne bloque déjà le même scénario).

## Architecture gelée — référentiel unique de vérité

**`docs/` est désormais l'unique source de vérité architecturale du projet.** L'architecture n'est plus à définir : elle est validée et figée dans ce référentiel. Toute nouvelle fonctionnalité doit s'y conformer intégralement ; tout écart nécessite une nouvelle ADR validée avant le code, jamais l'inverse. Point d'entrée : [docs/README.md](docs/README.md) — lire dans l'ordre `BUSINESS_RULES.md` → `DATA_DICTIONARY.md` → les ADR (`ADR_INDEX.md`) → `SYSTEM_ARCHITECTURE.md` → `modules/`+`api/` → `execution/EXECUTION_MASTER_PLAN.md` et les `SPRINT_XX.md`.

Le projet est désormais structuré en **13 modules fonctionnels** (voir [docs/modules/MODULES_INDEX.md](docs/modules/MODULES_INDEX.md)) — cette numérotation remplace l'ancien découpage `§5.x` du plan d'exécution historique (supprimé). Toujours citer le module concerné (nom + fichier spec, ex. `stay` / `docs/modules/stay.md`) dans les commits et PR.

En cas de conflit entre documents, **les specs de module détaillées (`docs/modules/*.md`) font foi sur les vues résumées** (`RBAC_MATRIX.md` tableau global, `audit-api.md`) — ces dernières peuvent être désynchronisées ou plus génériques que la spec détaillée.

## Règles non négociables

Ces règles doivent être respectées à chaque étape et rappelées dans chaque prompt de génération de module — formalisées dans les ADR ci-dessous, à consulter avant toute modification touchant leur périmètre :

1. Le **séjour (`Stay`)** est l'objet central opérationnel, décorrélé de la réservation. *([ADR-001](docs/ADR-001-Stay-Centric-Architecture.md))*
2. Un séjour peut avoir **plusieurs folios** (jamais « une réservation = une facture ») ; les lignes facturées sont immuables. *([ADR-002](docs/ADR-002-Folio-Billing-Model.md))*
3. La machine à états des chambres (`Room.statut`) est pilotée exclusivement par les événements de check-out, ménage (Gouvernante) et pannes bloquantes (Maintenance). *([ADR-003](docs/ADR-003-Room-State-Machine.md))*
4. Les paiements exigent une **idempotencyKey** unique ; les factures émises sont immuables ; montants en MAD. *([ADR-004](docs/ADR-004-Payment-Financial-Integrity.md))*
5. **Aucune suppression physique** (`DELETE`) sur les entités métier sensibles — soft delete (`deletedAt`) uniquement. Toute opération métier sensible doit appeler `AuditService.writeLog()` (`backend/src/modules/audit/audit.service.ts`) **dans la même transaction** que la modification métier — jamais après coup, jamais de manière asynchrone détachée — avec un motif écrit (≥ 10 caractères). S'applique à tout module futur touchant Payments, Folio, Stay, Room Transfer, Credit Note, etc. *([ADR-005](docs/ADR-005-Audit-Soft-Delete.md))*
6. Toute route d'écriture ou d'accès à des données sensibles est protégée côté serveur par `PermissionsGuard` selon la matrice fine de [docs/RBAC_MATRIX.md](docs/RBAC_MATRIX.md) — jamais de contrôle d'accès uniquement côté client. *([ADR-006](docs/ADR-006-RBAC-Enforcement.md))*
7. Le pointage RH (`TimeShiftSegment`) repose exclusivement sur l'horloge serveur ; une déconnexion pendant un shift actif est bloquée. *([ADR-007](docs/ADR-007-Time-Shift-Attendance.md))*
8. Les modules futurs se branchent sur les **services métier existants**, jamais en contournement (voir [docs/DEPENDENCY_GRAPH.md](docs/DEPENDENCY_GRAPH.md) pour la charte de dépendances autorisées entre modules).
